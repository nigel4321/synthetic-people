Reading each warning/fail in turn:

1. Parent peak RSS 1051 MB vs 500 MB threshold — FAIL by script.
   Parent RSS grew 5.7× while file size grew 46.5×. Sublinear, not
   file-size-proportional. If pyarrow.ipc.new_file were buffering
   the file before flushing, parent RSS would be ≈ 46 GB. It is
   ~2% of that. The 500 MB absolute threshold was written with the
   1 GB file in mind; the right test at 46 GB is "does parent RSS
   scale with file size?" — answer: yes, but at ~2% the rate.
   Streaming holds in the load-bearing sense.

   Likely cause: PyArrow memory-pool retention (jemalloc free-list
   keeps freed buffers rather than returning to OS) plus Python
   allocator high-water mark from per-batch numpy creation. Bounded
   growth, not buffering. Configurable via
   `pa.default_memory_pool().release_unused()` after each
   `write_batch`, or `PYARROW_DEFAULT_MEMORY_POOL=system`.

   Implication for n=1M: ~3000 batches per chromosome at 1024 sites;
   linear extrapolation suggests parent peak ≈ 6 GB. Tolerable on a
   32 GB host but worth flattening before 5d.1 lands.

   Action: small follow-up sub-spike before 5d.1 build — vary
   batch_size {256, 1024, 4096} × pool {jemalloc, system} on a
   moderate-scale run, identify the knob that flattens parent RSS.

2. Write throughput 157 MB/s vs 200 MB/s target — WARN by script.
   /tmp on this host is /dev/root (OS boot disk), not NVMe. At
   n=1M per-chrom Arrow file is ~80 GB; at 157 MB/s that's ~9 min
   per chromosome write. Across 22 chromosomes: ~3.3 hours of pure
   write I/O. Tolerable on a workstation; production runs at n=1M
   need NVMe (the plan already calls this out — this run is the
   empirical confirmation).

3. Aggregate read throughput 465 MB/s vs 500 MB/s target — WARN by
   script. 8 workers × 58 MB/s. Disk-bound: 46 GB file > 32 GB host
   RAM, so the page cache cannot hold the whole file and workers
   incur cold reads for the bulk of it. The small run's 1,355 MB/s
   came from 1 GB fitting fully in page cache. Same disk-type
   caveat as #2, not a 5d issue.

4. mmap-share PASS — the load-bearing test. System RAM delta
   +0.39 GB during the 8-worker phase, against a 372.56 GB
   worst-case if no sharing. Workers each report ~25 GB RSS while
   physical use is essentially zero — apparent-to-physical ratio
   ~520:1. The refcount-COW divergence that killed Phase 5e Phase A
   does not appear, even with 500k sites of realistic per-site
   Python iteration. This is the architectural bet for 5d and it
   survives the 50× scale-up cleanly.

Bottom line: 5d.1 green light unchanged. Two follow-ups identified:

  (a) Sub-spike to characterise parent-RSS vs (batch_size, pool)
      before 5d.1 build, so 5d.1 ships with a known-good default.
  (b) Tighten spike2's FAIL/WARN logic — the 500 MB absolute
      threshold should become a "growth rate vs file size" test so
      future big runs don't false-fail on monotone allocator
      high-water mark.
